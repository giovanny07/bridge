# Bridge

Bridge is a GLPI 11 plugin for migrating ITSM data from external platforms into
GLPI. The first supported source is SolarWinds Service Desk, formerly Samanage.

The plugin focuses on controlled migration: connection setup, source discovery,
dry-run validation, background job execution, migration history, rollback, and
traceability back to the source record.

## Compatibility

| Component | Requirement |
|-----------|-------------|
| GLPI | 11.0.0 to 11.0.99 |
| PHP | 8.1 or newer |
| PHP extensions | `curl` recommended |
| Current plugin version | 1.5.2 |

## Supported Data

| Source object | GLPI object | Status |
|---------------|-------------|--------|
| Incidents | Tickets | Supported |
| Service requests | Tickets with request type | Supported |
| Problems | Problems | Supported |
| Changes | Changes | Supported |
| Change tasks | Change tasks or change validations | Supported |
| Comments | ITIL followups | Supported |
| Attachments | GLPI documents linked to followups | Supported |
| Users | GLPI users and email records | Supported |
| Assets / ITAM | Not implemented |

SolarWinds source numbers are preserved in the GLPI title using the
`[ SD #number ]` prefix. GLPI object IDs remain generated by GLPI.

Then install and enable Bridge from **Setup > Plugins** in GLPI.

## GLPI Automatic Actions

Bridge creates background jobs and processes them through GLPI automatic actions.
The plugin registers these actions:

| Action | Purpose |
|--------|---------|
| `ProcessIncidents` | Processes incident and service request jobs |
| `ProcessChanges` | Processes change jobs |
| `ProcessProblems` | Processes problem jobs |
| `ProcessJobs` | Backward-compatible single-slot action |

The OS-level GLPI cron must run every minute. Example:

```cron
* * * * * apache /usr/bin/php /usr/share/glpi/front/cron.php
```

Use the same webserver user used by GLPI on the target system.

## Data Model

Bridge installs these tables:

| Table | Purpose |
|-------|---------|
| `glpi_plugin_bridge_connections` | Source system connection definitions |
| `glpi_plugin_bridge_migrations` | Per-record migration audit trail |
| `glpi_plugin_bridge_cursors` | Resumable pagination state |
| `glpi_plugin_bridge_jobs` | Background migration queue |
| `glpi_plugin_bridge_job_logs` | Per-chunk operational metrics and errors |

Runtime documents and temporary files are handled through GLPI storage paths, not
by writing into the plugin directory.

## Migration Flow

```text
Connection
   |
   v
ConnectorFactory
   |
   +--> SolarWindsClient reads source records, comments, tasks, attachments
   |
   v
SamanageNormalizer maps source fields to GLPI-shaped arrays
   |
   v
GlpiResolver matches entities, categories, groups, and users
   |
   v
MigrationEngine creates Ticket / Problem / Change
   |
   +--> creates followups
   +--> creates solutions
   +--> creates change tasks or validations
   +--> downloads and links documents when enabled
   |
   v
MigrationRecord stores audit and deduplication state
```

## Operating The Plugin

### Configure A Connection

1. Open **Setup > General Setup > Bridge**.
2. Create a connection.
3. Select the source system.
4. Enter the base URL.
5. Choose an authentication mode.
6. Configure fallback entity and fallback group.
7. Save and run **Test**.

For SolarWinds Service Desk, use the Samanage API base URL, for example:

```text
https://api.samanage.com
```

Bearer authentication is sent as `X-Samanage-Authorization: Bearer <token>`.

### Scan

Scan is read-only. It verifies that the API can return source records and gives a
small sample before migration.

### Dry-run

Dry-run does not write to GLPI. It validates:

- source record resolution
- entity matching
- category matching
- requester matching
- assignee user or group matching
- duplicate detection
- change task count for changes

Use dry-run before production migrations and export the CSV when mappings need
manual cleanup.

### Start Migration

Start migration creates a background job immediately. The job status page shows
progress while GLPI automatic actions process chunks.

Each job is scoped to one connection and one resource type. Incident, change, and
problem jobs can progress through their dedicated cron slots.

### Cancel, Retry, And Rollback

| Action | Behaviour |
|--------|-----------|
| Cancel | Marks a pending/running job as cancelled and stops the next chunk |
| Retry job | Creates a new job with the same options |
| Retry failed records | Purges failed audit records so they can be migrated again |
| Rollback | Deletes GLPI objects created by a completed job and clears their audit records |

Cancellation is cooperative: if a chunk is already executing, it finishes the
current PHP request and then stops the job from continuing.

## Source ID Handling

SolarWinds exposes both visible numbers and internal IDs. Bridge accepts either
form when migrating by source IDs:

```text
4682, 4780, 2957685
```

For changes and problems, visible numbers are resolved by scanning nearby API
pages before fetching the internal record.

## Mapping Rules

### Entity Matching

Entity names are matched with progressively relaxed normalization:

1. exact normalized name
2. accent-insensitive comparison
3. legal suffix cleanup
4. parenthetical suffix cleanup
5. abbreviation spacing and punctuation cleanup

If no entity matches, the connection fallback entity is used.

### Actors

| Source data | GLPI result |
|-------------|-------------|
| Requester email exists in GLPI | requester user linked |
| Requester email missing from GLPI | `alternative_email` stored |
| Requester missing | migration form fallback requester used |
| Assignee user email exists in GLPI | technician user linked |
| Assignee group name matches GLPI group | technician group linked |
| Assignee unresolved | connection fallback group used |

### SolarWinds Field Mapping

| SolarWinds | GLPI |
|------------|------|
| `number`, `name` | object title with `[ SD #number ]` prefix |
| `description_no_html` | content |
| `state` | GLPI status |
| `priority` | GLPI priority |
| `origin` | request type |
| `site.name` | entity |
| `category.name`, `subcategory.name` | ITIL category |
| `created_at` | opening and creation date |
| `updated_at` | solved/closed date when applicable |
| comments | ITIL followups |
| attachments | GLPI documents |
| change tasks | ChangeTask or ChangeValidation |

## Performance Controls

| Setting | Default | Notes |
|---------|---------|-------|
| Migration limit | 50 | Maximum records requested for a job |
| `CHUNK_PAGES` | configured in `BridgeJobConfig` | Pages processed per cron tick |
| `parallel_api_pages` | 1 | Pages fetched concurrently per wave |
| `PARALLEL_API_MAX` | configured in `BridgeJobConfig` | Hard cap for API concurrency |
| Rate-limit backoff | configured in `BridgeJobConfig` | Sleep before retrying HTTP 429 |

Recommended tuning:

1. Keep `parallel_api_pages = 1` for first production migrations.
2. Increase to 2 or 4 only after reviewing job logs.
3. Keep attachments disabled for initial validation runs.
4. Migrate by source ID for spot checks before large date ranges.
5. Use dry-run CSV to clean mappings before production batches.

## Extension Points

To add a new source system:

1. Implement `ConnectorInterface`.
2. Implement `NormalizerInterface`.
3. Register both in `ConnectorFactory`.
4. Add the system type to `Connection::getSupportedSystems()`.
5. Add unit tests for connector behaviour and field mapping.
6. Add API contract tests if a live sandbox is available.

The migration engine, resolver, job queue, history, and UI are shared across
connectors.

## Troubleshooting

| Symptom | Check |
|---------|-------|
| Job stays pending | GLPI automatic actions and OS cron |
| Cancel button does not respond | Browser console, AJAX response, CSRF token |
| Source ID returns 404 | Use visible number or internal ID; verify base URL |
| UI changes not visible | GLPI compiled template cache and asset cache |
| Attachments fail | PHP temp directory, document upload permissions, API auth |
| Records go to wrong entity | Dry-run CSV and fallback entity configuration |
| Slow start | Ensure Start migration creates a job directly; use dry-run separately |
